## Findings Considered And Rejected

- Retrying failed provider/runtime executions in the first worker slice: rejected for now because `agent_schedule_runs.agent_run_id` is currently one-to-one with a generic `AgentRun`, and terminal `AgentRun` rows cannot be restarted. This plan retries claim/setup failures before a generic run starts, and marks actual runtime failures terminal so users can inspect and re-enable deliberately.

### From the 2026-07-01 Pydantic AI integration audit

- **Adopt `pydantic-ai-harness` / CodeMode now**: deferred, not planned. The
  harness is not installed, the old demo tools have been removed, and the
  configurable runtime catalog is still too small to justify a sandbox layer.
  Revisit when a real multi-tool catalog or MCP toolsets land; per
  `docs/pydantic-ai/13-advanced-and-ecosystem.md` it must be
  per-workspace opt-in behind approval. Caveat recorded for that future work:
  approval-required tools are **excluded** from the Monty sandbox, so in an
  approval-heavy workspace CodeMode's benefit shrinks. Harness v0.5.0 is
  compatible with the pinned pydantic-ai 2.1.0.
- **Capabilities-on-demand / ToolSearch (`defer_loading`)**: deferred until the
  tool catalog is large enough to bloat prompts. The plumbing already exists
  (`RuntimeToolDefinition.defer_loading` in `runtime/tools/contract.py`).
- **`FallbackModel` provider failover**: not planned; transport retries
  (plan 010) cover transient failures, and multi-provider failover changes
  billing/behavior semantics that need a product decision first.
- **Rebuilding the conversation-naming `Agent` per call**
  (`services/conversations/naming.py:58`): not worth doing — construction is
  cheap, the model can vary by settings, and caching adds state for no
  measurable win.
- **Migrating the SSE driver or approval flow**: nothing to do — verified that
  `run_stream_events()` + `DeferredToolRequests`/`DeferredToolResults` +
  `requires_approval` usage matches current 2.x guidance exactly; the code
  already uses the 2.x event shapes (`FunctionToolResultEvent.part`,
  `result.usage` as a property).

### From the 2026-07-01 skills planning pass

- **Provider-native file APIs for skill documents**: rejected by product
  decision — Praxis owns document storage (provider-neutral `StorageProvider`,
  plan 002) and converts to markdown itself, so skills behave identically
  across model providers.
- **A new `skill.activated` SSE event**: rejected for now. The frontend SSE
  parser throws on unknown event names (`stream/sse.ts`), so a new event
  breaks stale clients mid-deploy; activations already flow through the
  existing `tool.call`/`tool.result` events (`name == "load_capability"`,
  persisted parts carry `tool_kind == "capability-load"`). Plan 020 renders
  from those. If a dedicated event is ever needed, ship the client whitelist
  change first.
- **`script_refs` execution for skills**: still deferred — requires the
  sandbox story (see the CodeMode note above). The column stays commented out
  in `models/skills.py`.
- **Semantic/vector skill matching (pgvector)**: not planned — catalogs are
  small and pydantic-ai's `load_capability` routing off name+description is
  the standard mechanism (matches the open Agent Skills standard's level-1
  metadata design). Revisit only if workspaces grow to dozens of skills and
  models mis-route.
- **Skill import/export as SKILL.md bundles**: not planned yet, but the name /
  description / instructions field rules in plan 016 deliberately match the
  open standard so a future importer/exporter is mechanical.
